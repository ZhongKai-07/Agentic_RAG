**主要问题**
- `高` 权限边界基本没落地。空间创建/详情/改规则不读 `X-User-Id`，[SpaceController.java](E:/AI%20Application/Agentic_RAG/rag-adapter-inbound/src/main/java/com/rag/adapter/inbound/rest/SpaceController.java#L27)；文档列表/详情/批量删改也不校验用户，[DocumentController.java](E:/AI%20Application/Agentic_RAG/rag-adapter-inbound/src/main/java/com/rag/adapter/inbound/rest/DocumentController.java#L57)；会话详情/删除同样可越权访问，[ChatController.java](E:/AI%20Application/Agentic_RAG/rag-adapter-inbound/src/main/java/com/rag/adapter/inbound/rest/ChatController.java#L56)。这对“企业级”是当前最大的上线阻塞项。
- `高` 新建空间后，创建者很可能在列表里看不到它。创建空间时没有写任何访问规则，[SpaceApplicationService.java](E:/AI%20Application/Agentic_RAG/rag-application/src/main/java/com/rag/application/identity/SpaceApplicationService.java#L26)；而可见空间查询完全依赖 access rule，[AccessRuleJpaRepository.java](E:/AI%20Application/Agentic_RAG/rag-adapter-outbound/src/main/java/com/rag/adapter/outbound/persistence/repository/AccessRuleJpaRepository.java#L14)。这会直接造成“创建成功但列表消失”的产品级故障。
- `高` 新 space 首次入库有失败风险。索引前先删旧 chunk，[IndexEventHandler.java](E:/AI%20Application/Agentic_RAG/rag-application/src/main/java/com/rag/application/event/IndexEventHandler.java#L72)；但本地 OpenSearch 只有 upsert 时才确保 index 存在，[LocalOpenSearchAdapter.java](E:/AI%20Application/Agentic_RAG/rag-adapter-outbound/src/main/java/com/rag/adapter/outbound/vectorstore/LocalOpenSearchAdapter.java#L34)；删除接口本身不兜底不存在的 index，[LocalOpenSearchAdapter.java](E:/AI%20Application/Agentic_RAG/rag-adapter-outbound/src/main/java/com/rag/adapter/outbound/vectorstore/LocalOpenSearchAdapter.java#L66)。第一次上传到新索引时，流程可能先被 `deleteByQuery` 打成 `FAILED`。
- `高` SSE 引用事件前后端协议不一致。后端发的是 `CitationEmit(Citation citation)`，JSON 会多一层 `citation`，[StreamEvent.java](E:/AI%20Application/Agentic_RAG/rag-domain/src/main/java/com/rag/domain/conversation/model/StreamEvent.java#L14)；前端却按扁平字段解析，[useSSE.ts](E:/AI%20Application/Agentic_RAG/rag-frontend/src/hooks/useSSE.ts#L117)。结果是流式回答里的引用大概率收不到，右侧 citation 面板会空。
- `中` 上传链路对大文件不友好。控制器直接 `file.getBytes()`，[DocumentController.java](E:/AI%20Application/Agentic_RAG/rag-adapter-inbound/src/main/java/com/rag/adapter/inbound/rest/DocumentController.java#L31)；批量上传也是同样模式，[DocumentController.java](E:/AI%20Application/Agentic_RAG/rag-adapter-inbound/src/main/java/com/rag/adapter/inbound/rest/DocumentController.java#L43)。100MB 上限加 batch upload 会把多个文件一次性灌进 JVM 堆。解析阶段拿到的文件流也没关闭，[ParseEventHandler.java](E:/AI%20Application/Agentic_RAG/rag-application/src/main/java/com/rag/application/event/ParseEventHandler.java#L56)。
- `中` 删除文档只清理当前版本文件，历史版本会残留在存储里。[DocumentApplicationService.java](E:/AI%20Application/Agentic_RAG/rag-application/src/main/java/com/rag/application/document/DocumentApplicationService.java#L92) 和 [DocumentApplicationService.java](E:/AI%20Application/Agentic_RAG/rag-application/src/main/java/com/rag/application/document/DocumentApplicationService.java#L101) 都只处理 `currentVersion.filePath()`，数据库会级联删版本记录，但磁盘文件不会。
- `中` 流式聊天的资源回收不完整。控制器里直接 `subscribe()`，[ChatController.java](E:/AI%20Application/Agentic_RAG/rag-adapter-inbound/src/main/java/com/rag/adapter/inbound/rest/ChatController.java#L89)，但没有 `emitter.onCompletion/onTimeout` 去 cancel；客户端断开后，底层 LLM 流和线程仍可能继续跑。
- `中` Agent 失败降级过于乐观。planner 解析失败会 fallback 到空 query，[LlmRetrievalPlanner.java](E:/AI%20Application/Agentic_RAG/rag-application/src/main/java/com/rag/application/agent/LlmRetrievalPlanner.java#L108)；evaluator 解析失败则直接当“足够可答”继续生成，[LlmRetrievalEvaluator.java](E:/AI%20Application/Agentic_RAG/rag-application/src/main/java/com/rag/application/agent/LlmRetrievalEvaluator.java#L101)。这类故障不会显性报错，而是静默地产出低质量答案。
